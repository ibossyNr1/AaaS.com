1. The Autoresearch Pattern — Distilled

Karpathy's Core Insight

Autoresearch gives an AI agent a real system, lets it modify one file, runs for a fixed time budget, measures one metric (val_bpb), and applies a binary decision: if improved → keep (git commit), if equal or worse → discard (git reset). Repeat indefinitely. The human sleeps while the agent runs ~100 experiments overnight.

The Three-File Architecture

prepare.py Human-controlled

Constants, data preparation, runtime utilities. The fixed environment. The agent cannot touch this.

train.py Agent-modified

The model, optimizer, training loop. The single file the agent mutates. All experiments are diffs to this file.

program.md Human-controlled

Agent instructions and research directives. The "research org code" — tells the agent what to explore, what constraints to respect, and how to evaluate.

The Generalization (AutoExp)

The community has already generalized this into AutoExp — autonomous experimentation for any quantifiable metric. The pattern works whenever three conditions hold:

1. Measurable Fitness Signal

You can define a quantitative metric: conversion rate, latency, defect rate, revenue lift, content engagement, framework alignment score, etc.

2. Repeatable Experiment

You can run a controlled experiment with a fixed time/resource budget, producing comparable results each time.

3. Automatic Survival Decision

The system can decide what survives (improved) and what gets reverted (equal or worse) — without human intervention.

2. The Integration: Autoresearch × The 11-Step Protocol

The key insight is this: every step in your 11-step protocol already produces measurable artifacts. The autoresearch pattern can be layered on top so that each step doesn't just produce output — it produces output, measures it, and iteratively improves it before advancing to the next step.

Two Levels of Integration

Level 1: Intra-Step Optimization

Within each step, the agent runs an autoresearch loop on its own output. For example, Step 9 (Context Synthesis) doesn't just produce the CLIENT_CONTEXT_FRAMEWORK.md once — it produces it, measures the completeness scores, mutates its synthesis approach, re-runs, and keeps the version with the highest scores. Each step becomes self-optimizing.

Per-step loops

Level 2: Cross-Step Optimization

Across the entire pipeline, the system tracks which experiment variants at each step produced the best downstream results. Over multiple project runs, the pipeline learns which prompt templates, skill chains, and research configurations yield the highest-quality final deliverables. The program.md for the entire protocol evolves.

Pipeline-level learning

3. The KPI Auto-Definition Layer

This is the critical new architecture component. Every artifact produced by every step must auto-define its own KPIs — the measurable fitness signals that the autoresearch loop will optimize against.

The ARTIFACT_KPI Schema

Every output document gets a KPI block appended (or tracked in a parallel _kpi_state.json):

4. The Three-File Pattern Applied to Each Step

Karpathy's genius is in the constraint: three files, one mutable. We apply this same pattern to every step in the protocol.

Autoresearch RoleOriginal (ML)AaaS Protocol EquivalentWhat It Does
prepare.py
(fixed environment)		 Data prep, constants All upstream artifacts
(GOAL_BRIEF.md, RESEARCH_BRIEF.md, etc.)		 The immutable inputs. The agent reads these but cannot modify them. They are the "dataset."
train.py
(agent-mutated)		 Model + training loop The step's execution script / prompt template
(e.g., context_intelligence_engine's synthesis prompt)		 The single file the agent experiments on. It modifies the approach, reruns, measures, keeps or discards.
program.md
(human-directed)		 Research instructions The step's SKILL.md / project.json instructions
(e.g., active_reasoning's 4-phase spec)		 The human-authored instructions that tell the agent what to explore and what constraints to respect.

Example: Step 9 (Context Synthesis) as an Autoresearch Loop

prepare.py equivalent: The 6 upstream documents (GOAL_BRIEF.md through INDUSTRY_KNOWLEDGE_BASE.md) — read-only inputs.

train.py equivalent: The synthesis prompt template + the RAG query strategy in context_intelligence_engine. The agent mutates: chunk size, embedding strategy, section-merging logic, cross-reference density, summarization vs. extraction balance.

program.md equivalent: The project.json instructions for context_intelligence_engine — the human-authored directives.

Metric (val_bpb equivalent): avg_section_completeness — the mean of completeness scores across sections 1.0–5.0.

Budget: 10 iterations. Each iteration runs the full synthesis, scores the output, and keeps or discards.

Result: Instead of one synthesis pass, the agent autonomously tries 10 different approaches and keeps the best one. The human wakes up to a framework document that's been self-optimized.

5. Pipeline-Level Learning (Cross-Project Memory)

This is where it gets powerful. The autoresearch loop doesn't just optimize within one project — it learns across ALL projects.

The Pipeline Evolution Loop

After every completed project, the system records the final KPI scores for all 11 steps plus the client satisfaction score (the ultimate downstream metric). Over time, it builds a dataset:

6. Implementation Architecture

New Components to Build

1. KPI Extractor Agent

A new skill that runs after every step's output and automatically computes the KPI scores from the artifact content. Uses the scoring rubrics defined in the KPI schema.

Input: Any artifact .md file + the step's KPI definition

Output: ARTIFACT_KPI JSON block

New skill needed

2. Experiment Runner

A wrapper around each step that implements the autoresearch loop: run step → extract KPIs → compare to best → keep or discard → repeat N times.

Input: Step config + iteration budget + upstream artifacts

Output: Best-scoring artifact + experiment log

New skill needed

3. Pipeline Memory Store

A Firestore collection (or SQLite in GravityClaw) that persists experiment logs across all projects. Enables cross-project learning.

Schema: project_id, step, variant_hash, kpi_scores, downstream_outcomes

Firestore / SQLite

4. Evolution Engine

Periodic (weekly/monthly) process that analyzes the pipeline memory, identifies winning variants, and promotes them to defaults. Updates the program.md equivalents (project.json instructions) for each step.

Trigger: Scheduled via GravityClaw scheduler

GravityClaw module

7. Concrete KPI Definitions — How Each Asset Auto-Defines Them

The crucial design decision: KPIs are not manually specified per project. They are auto-defined by asset type. Every template in the master library carries a KPI specification.

Template-Level KPI Registry

Each template in drive_structure_builder/master_templates/ gets a companion _kpi.json:

master_templates/30-39_Growth_Marketing/30.01_Marketing_Strategy/ ├── Comprehensive_Marketing_Plan_&_Budget.md ← the template └── Comprehensive_Marketing_Plan_&_Budget._kpi.json ← auto-defined KPIs _kpi.json contents: { "template": "Comprehensive_Marketing_Plan_&_Budget", "department": "30-39_Growth_Marketing", "primary_kpi": { "name": "section_completeness", "method": "count_filled_sections / total_sections", "target": 0.9 }, "secondary_kpis": [ { "name": "budget_table_populated", "method": "has_numeric_values_in_budget_section", "target": true }, { "name": "timeline_specificity", "method": "count_dated_milestones", "target": 5 }, { "name": "channel_coverage", "method": "count_distinct_channels_mentioned", "target": 4 }, { "name": "word_count", "method": "total_words", "target": 2000, "direction": "range", "min": 1500, "max": 4000 } ], "autoresearch": { "iteration_budget": 5, "mutable_dimensions": ["tone", "depth_per_section", "example_density", "data_citation_frequency"] } }

Auto-Definition Rules

For templates that don't yet have a _kpi.json, the KPI Extractor agent auto-generates one by analyzing the template structure:

Template PatternAuto-Inferred KPIs

The Compounding Effect

This is the lock-on. Every project makes every subsequent project better. The system literally gets smarter with every client. Karpathy's autoresearch runs ~100 experiments overnight on one ML training setup. Your system runs ~50–100 experiments per step × 11 steps × every project, all feeding a shared pipeline memory that optimizes the entire protocol.

After 10 projects, your Step 9 synthesis isn't running the generic approach — it's running the variant that emerged as the winner across 10 projects' worth of autonomous experimentation. After 50 projects, the entire protocol is self-optimized to a degree no human could achieve through manual iteration.

The Bottom Line

Karpathy built autoresearch for one metric (val_bpb) on one file (train.py) in one domain (ML training). Your system generalizes this to 11 metrics across 11 steps across every business domain your templates cover. The three-file pattern (fixed inputs / agent-mutable approach / human-directed instructions) maps perfectly onto your DOE 6-layer architecture — where Layer 1 (Directive) is program.md, Layer 6 (Execution) is train.py, and Layers 2–3 (Context + Knowledge) are prepare.py.

The result: a protocol that doesn't just produce assets — it produces assets, measures them, experiments with alternatives, keeps the best, and learns across every project it ever runs. This is the difference between a workflow and an intelligence.
